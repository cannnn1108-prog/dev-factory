# 大規模プロジェクト戦略マニュアル

Claude Code で大規模プロジェクトを中断せずに作りきるための戦略。

---

## なぜ途中で止まるのか

Claude Code でプロジェクトが途中で止まる主な原因は3つ。

### 1. コンテキストウィンドウの圧迫

1回のセッションで全画面を作ろうとすると、途中でコンテキストが埋まって精度が落ちる、または止まる。

### 2. 1回の指示が大きすぎる

「全ページ作って」は処理量が多すぎて途中で切れる。

### 3. エラーの連鎖

1箇所のエラーを直そうとして別のエラーが出て、無限ループに入る。

---

## 対策1: 1セッション = 1画面を徹底する

1回のプロンプトで1画面だけを完成させる。

```
セッション1: プロジェクト初期化 + フォルダ構成
セッション2: 共通レイアウト + サイドバー
セッション3: ダッシュボード画面
セッション4: AI投稿生成画面
セッション5: 予約投稿画面
...
```

### 運用ルール

- 各セッション終了時に `git commit` して区切る
- 次のセッションでは「前回○○まで完成しました。次は△△ページを作ってください」と伝える
- 前のコンテキストを引き継がずに済む

---

## 対策2: CLAUDE.md にプロジェクトルールを書く

プロジェクトルートに `CLAUDE.md` を置いて、毎セッション自動で読み込ませる。

```markdown
# {プロジェクト名}

## 技術スタック
- Next.js 14 App Router / TypeScript / Tailwind CSS

## デザインルール
- 背景: ネイビー〜ブラック (#0a0b1a)
- カード: 半透明ダーク + 青紫発光ボーダー
- アイコン: Lucide React
- 角丸: rounded-2xl
- 余白: 広め (p-6, gap-6)

## フォルダ構成
- src/app/(dashboard)/  — 各ページ
- src/components/ui/    — 共通UIコンポーネント
- src/components/layout/ — Sidebar, Header
- src/lib/              — DB接続, API呼び出し
- src/types/            — 型定義

## 完了済み
- [x] プロジェクト初期化
- [x] 共通レイアウト
- [x] ダッシュボード
- [ ] AI投稿生成画面  ← 次はここ
```

**ポイント**: 「完了済み」リストを毎回更新すれば、Claude Code は現在地を正確に把握できる。

---

## 対策3: /compact で途中圧縮する

1セッション内で長くなってきたら、チャット欄に `/compact` と入力する。

- 会話履歴を要約圧縮してコンテキストを空ける
- 同一セッション内でも作業を続けやすくなる

---

## 対策4: エラーループを防ぐ指示の書き方

すべての指示に以下を付与する。

```
* エラーが出たら3回まで自力で修正してください
* 3回で直らなければ、何が起きているか報告して止まってください
* 既存ファイルは壊さないでください
* 新しいファイルを作る前にパスを教えてください
```

これだけで無限ループで止まるパターンが大幅に減る。

---

## 対策5: 段階的に複雑度を上げる

| Phase | 内容 | DB | API |
|-------|------|----|----|
| Phase 1 | 全画面のUI（ダミーデータ） | なし | なし |
| Phase 2 | DB接続 + CRUD | あり | なし |
| Phase 3 | 外部API連携（Anthropic API等） | あり | あり |
| Phase 4 | 予約投稿のcron + 外部API連携 | あり | あり |

**原則**: Phase 1 を全画面分完成させてから Phase 2 に進む。縦に深掘りせず、横に広げてから深くする。

---

## 対策6: ヘッドレスモード / Agent SDK で自動化する（上級）

Claude Code のヘッドレスモード（`claude -p "指示"` でCLI実行）を使えば、シェルスクリプトで順番に実行できる。

```bash
claude -p "src/app/(dashboard)/generate/page.tsx を作成して。AI投稿生成画面のUIをダミーデータで実装。デザインはCLAUDE.mdに従う"

claude -p "src/app/(dashboard)/schedule/page.tsx を作成して。予約投稿画面のUIをダミーデータで実装。デザインはCLAUDE.mdに従う"
```

各コマンドが独立したセッションなので、コンテキスト圧迫の問題がなくなる。

---

## 実践ワークフロー: まとめ

最も確実な進め方を以下に示す。

### 毎セッションのルーティン

```
1. CLAUDE.md にルールと進捗を書く
2. 1セッション1画面で進める
3. 各画面完成後に git commit
4. CLAUDE.md の完了リストを更新
5. 次のセッションで次の画面に着手
6. 長くなったら /compact
```

### プロジェクト全体の推奨順序

```
1. Claude Code をセットアップ
2. フレームワークで土台を作る（Next.js等）
3. メイン画面のUIを再現する
4. サブ画面を順次追加する
5. DB保存を入れる
6. AI API連携で核心機能を実装
7. 予約・自動化機能を作る
8. 外部API連携を足す
9. デプロイする
```

### 成功の鍵

> 「一気に全部作る」ではなく「10回に分けて確実に積み上げる」

この順番なら、途中で止まっても「見た目の完成度が高い使えるアプリ」が残るので、挫折しにくい。

## Claude Code 活用のベストプラクティス

### スキル機能の活用

`.claude/skills/` にプロジェクト専用ルールを保存して、毎回の指示を簡略化する。

例: `.claude/skills/ui-saas-dark/SKILL.md`

```markdown
* ダークネオンで作る
* 余白広め
* 角丸大きめ
* 青紫発光
* SaaS風の高級感
* Lucide icons
* 再利用可能コンポーネント優先
```

### 指示の具体性

| NG | OK |
|----|-----|
| 「かっこよくして」 | 「背景を#0a0b1a、カードに青紫発光ボーダー、角丸rounded-2xl」 |
| 「なんか違う」 | 「余白を増やす、文字サイズの強弱をつける、サイドバーの選択中を目立たせる」 |
| 「いい感じにして」 | 改善点を箇条書きで5つ以上列挙 |
| 「全部作って」 | 「ダッシュボード画面だけ先に完成させて」 |

### 外部API連携の段階的アプローチ

外部API（X API等）は最初から本番連携に突っ込まず、まずローカル実装で完結させる。

```
ステップ1: UIだけ作る（ダミーデータ）
ステップ2: ローカルDBに保存する
ステップ3: ステータス管理（draft / scheduled / posted）
ステップ4: 最後にAPIを繋ぐ
```

これで「ツールとして動いてる感」を早期に出せる。
