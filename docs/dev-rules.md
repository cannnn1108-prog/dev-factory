# 開発ルール・テンプレート

dev-factory リポジトリにおける開発ルールと標準テンプレートを定義する。

---

## 1. ブランチ戦略

| ブランチ | 用途 | マージ先 |
|---------|------|---------|
| `main` | 本番リリース用。常に安定状態を保つ | — |
| `develop` | 開発統合ブランチ | `main` |
| `feature/<名前>` | 新機能開発 | `develop` |
| `fix/<名前>` | バグ修正 | `develop` |
| `hotfix/<名前>` | 緊急修正 | `main` + `develop` |
| `claude/<タスク名>` | Claude Code による自動開発 | `develop` or `main` |

### ブランチ命名規則

```
feature/add-user-auth
fix/login-validation-error
hotfix/critical-api-crash
claude/implement-dashboard-MBcei
```

- 英語小文字、ハイフン区切り
- 簡潔かつ内容が分かる名前にする

---

## 2. コミットメッセージ規約

### フォーマット

```
<type>: <概要>

<本文（任意）>
```

### Type 一覧

| Type | 用途 |
|------|------|
| `feat` | 新機能追加 |
| `fix` | バグ修正 |
| `docs` | ドキュメントのみの変更 |
| `style` | コードの意味に影響しない変更（フォーマット等） |
| `refactor` | リファクタリング |
| `test` | テストの追加・修正 |
| `chore` | ビルドプロセスや補助ツールの変更 |

### 例

```
feat: ユーザー認証機能を追加

OAuth2.0を使用したGoogle認証を実装。
セッション管理にはJWTトークンを使用する。
```

---

## 3. コードレビュー規約

### レビュー基準

- [ ] 機能要件を満たしているか
- [ ] テストが追加・更新されているか
- [ ] セキュリティ上の問題がないか（OWASP Top 10）
- [ ] パフォーマンスへの影響が許容範囲か
- [ ] コーディング規約に従っているか
- [ ] ドキュメントが更新されているか（必要な場合）

### レビュープロセス

1. PR 作成者がセルフレビューを実施
2. 最低 1 名のレビュアーの Approve が必要
3. CI が全て通過していること
4. コンフリクトが解消されていること

---

## 4. ディレクトリ構成テンプレート

新しいプロジェクトを追加する際の標準構成：

```
projects/<project-name>/
├── README.md           # プロジェクト概要・セットアップ手順
├── src/                # ソースコード
├── tests/              # テストコード
├── docs/               # プロジェクト固有ドキュメント
├── scripts/            # ビルド・デプロイスクリプト
├── .env.example        # 環境変数テンプレート（秘密情報は含めない）
└── Makefile            # タスクランナー（任意）
```

## 5. コーディング規約

### 共通ルール

- インデント: スペース 2 つ（言語固有の慣例がある場合はそちらに従う）
- ファイル末尾: 改行 1 つ
- 文字コード: UTF-8
- 改行コード: LF

### 言語別ルール

#### Python
- フォーマッター: `ruff format`
- リンター: `ruff check`
- 型ヒント: 必須（公開関数・メソッド）
- Python バージョン: 3.11+

#### TypeScript / JavaScript
- フォーマッター: `prettier`
- リンター: `eslint`
- `strict` モード有効
- Node.js バージョン: 20 LTS+

#### その他の言語
- 各言語のデファクトスタンダードに従う
- プロジェクト README に使用ツールを明記する

---

## 6. セキュリティルール

### 禁止事項

- シークレット情報（API キー、パスワード等）をコードにハードコードしない
- `.env` ファイルをコミットしない（`.gitignore` に必ず追加）
- 本番データをローカル環境にコピーしない

### 必須事項

- 外部入力は必ずバリデーションする
- SQL はパラメータ化クエリを使用する
- 依存パッケージは定期的にアップデートし、脆弱性をチェックする
- HTTPS を使用する（HTTP は開発環境のみ）

---

## 7. テスト規約

### テスト種別

| 種別 | 対象 | 必須/推奨 |
|------|------|----------|
| ユニットテスト | 個別の関数・メソッド | 必須 |
| 統合テスト | モジュール間の連携 | 推奨 |
| E2E テスト | ユーザーフロー全体 | 機能による |

### テスト命名規則

```
test_<対象>_<条件>_<期待結果>
```

例: `test_login_invalid_password_returns_401`

---

## 8. CI/CD ルール

### マージ前チェック（必須）

- リンター通過
- テスト全件通過
- ビルド成功
- セキュリティスキャン通過

### デプロイ

- `main` へのマージで本番デプロイ（自動 or 手動承認）
- デプロイ前にステージング環境で確認
- ロールバック手順を常に用意する

---

## 9. ドキュメントルール

### 必須ドキュメント

各プロジェクトには以下を含めること：

1. **README.md** — プロジェクト概要、セットアップ手順、使い方
2. **CHANGELOG.md** — バージョンごとの変更履歴（任意だが推奨）
3. **API ドキュメント** — 外部公開 API がある場合

### ドキュメント更新タイミング

- 機能追加・変更時に同時更新
- 「後で書く」は禁止 — PR に含める

## 10. Issue / PR テンプレート

### Issue テンプレート

```markdown
## 概要
<!-- 何をしたいか / 何が問題か -->

## 詳細
<!-- 再現手順、期待動作、実際の動作など -->

## 受け入れ条件
- [ ] 条件1
- [ ] 条件2
```

### PR テンプレート

```markdown
## 変更内容
<!-- 何を変更したか -->

## 関連 Issue
closes #XX

## テスト
- [ ] ユニットテスト追加/更新
- [ ] 動作確認済み

## スクリーンショット（UI変更がある場合）
```
