# 開発ルール・テンプレート

dev-factory リポジトリにおける開発ルール、Claude Code 活用パターン、プロンプトテンプレートを定義する。

---

## 1. プロジェクト初期化テンプレート

新しいプロジェクトを始める際、Claude Code に最初に渡すべき指示のテンプレート。

### 初回プロンプトテンプレート

```
「{プロジェクト名}」のWebアプリを作りたいです。
初心者でも保守しやすい構成にしてください。

要件:
* 技術スタック（例: Next.js App Router / TypeScript / Tailwind CSS）
* UIテーマ（例: ダークテーマ、青紫のネオン / グラスモーフィズム風UI）
* 画面一覧（例: ダッシュボード、設定画面、etc.）
* まずはダミーデータで動くMVPを作る
* コードはコンポーネント分割する
* 将来のAPI/DB追加を見据えた構成にする

最初にやってほしいこと:
1. フォルダ構成を提案
2. 実装計画を短く出す
3. メイン画面のUIから着手
4. 変更前にどのファイルを作るか教える
```

**ポイント**: 「何を作るか」だけでなく「どういう順番で作るか」まで指定する。Claude Code 公式でも、コードベース理解 → コード変更の順が推奨されている。

## 2. UI仕様の渡し方

デザインの再現度を上げるには、見た目の要素を仕様書にして渡す。

### UI仕様テンプレート

```
このUIの特徴を反映してください。

画面仕様:
* 背景色（例: 濃いネイビー〜ブラック）
* カードスタイル（例: 半透明暗色、青〜紫の発光ボーダー）
* レイアウト（例: 左サイドバー固定、メイン領域にカード配置）
* 角丸サイズ（例: 大きめ rounded-2xl）
* アイコンライブラリ（例: Lucide）
* 余白（例: 広め p-6, gap-6）
* 雰囲気（例: SaaS風、高級感）
* レスポンシブ対応
```

**コツ**: 「かっこよく」ではなく「黒」「青紫」「発光」「グラス」「大きめ角丸」「高級感」まで具体的に言う。

## 3. 段階的開発ルール

### 原則: 1セッション = 1画面

全機能を一度に作らせない。以下の順序で段階的に進める。

#### Phase 1: UIのみ（ダミーデータ）

| 順番 | 作業内容 | DB | API |
|------|---------|----|----|
| 1 | プロジェクト初期化 + フォルダ構成 | なし | なし |
| 2 | 共通レイアウト + サイドバー | なし | なし |
| 3 | メイン画面（ダッシュボード等） | なし | なし |
| 4 | サブ画面を順次追加 | なし | なし |

#### Phase 2: データ保存

| 順番 | 作業内容 | DB | API |
|------|---------|----|----|
| 5 | DB接続（Supabase等） | あり | なし |
| 6 | テーブル設計 + CRUD | あり | なし |

#### Phase 3: AI機能

| 順番 | 作業内容 | DB | API |
|------|---------|----|----|
| 7 | Anthropic API 連携 | あり | あり |
| 8 | プロンプト設計・調整 | あり | あり |

#### Phase 4: 外部連携

| 順番 | 作業内容 | DB | API |
|------|---------|----|----|
| 9 | 外部API連携（X API等） | あり | あり |
| 10 | デプロイ | あり | あり |

**重要**: 横に広げてから深くする。縦に深掘りしない。

## 4. Claude Code への指示テンプレート集

### 画面作成指示

```
{画面名}画面だけ先に完成させてください。

必要なこと:
* {UI要素のリスト}
* ダークネオンUI
* ダミーデータでOK

また、共通レイアウトとして再利用できるようにしてください。
```

### UI修正指示

```
今のUIをブラッシュアップしてください。

改善点:
* {具体的な改善点を箇条書き}
```

**注意**: 「なんか違う」で終わらせない。何が違うのかを箇条書きで言う。

### ページ追加指示

```
今の共通レイアウトを使って、以下のページを追加してください。
* /path1
* /path2
* /path3

それぞれ仮のUIでよいですが、あとで機能追加しやすいように
コンポーネント分割してください。
```

### DB導入指示

```
このアプリに{DB名}を導入したいです。
以下のテーブル設計を提案し、SQLも作ってください。

必要テーブル:
* table1
* table2

やってほしいこと:
1. 各テーブルのカラム設計
2. 型定義
3. DB接続用ユーティリティ
4. ダミーデータを実データに差し替えやすい形に変更
```

### AI機能追加指示

```
Anthropic APIを使って、{機能名}を追加してください。

要件:
* 入力項目: {リスト}
* 出力項目: {リスト}
* API Routeで実装
* APIキーは環境変数管理
* フロントには生成中ローディングを表示
* 結果はカード表示
* 後から保存できるようにする
```

### エラーループ防止指示（全指示に付与推奨）

```
* エラーが出たら3回まで自力で修正してください
* 3回で直らなければ、何が起きているか報告して止まってください
* 既存ファイルは壊さないでください
* 新しいファイルを作る前にパスを教えてください
```

## 5. CLAUDE.md テンプレート

プロジェクトルートに配置し、毎セッション自動読み込みさせる。

```markdown
# {プロジェクト名}

## 技術スタック
- Next.js 14 App Router / TypeScript / Tailwind CSS

## デザインルール
- 背景: {色指定}（例: #0a0b1a）
- カード: {スタイル}（例: 半透明ダーク + 青紫発光ボーダー）
- アイコン: {ライブラリ}（例: Lucide React）
- 角丸: {サイズ}（例: rounded-2xl）
- 余白: {サイズ}（例: p-6, gap-6）

## フォルダ構成
- src/app/(dashboard)/  — 各ページ
- src/components/ui/    — 共通UIコンポーネント
- src/components/layout/ — Sidebar, Header
- src/lib/              — DB接続, API呼び出し
- src/types/            — 型定義

## 完了済み
- [x] プロジェクト初期化
- [x] 共通レイアウト
- [ ] ダッシュボード  ← 次はここ
```

**運用ルール**: 各セッション終了時に「完了済み」リストを更新する。

## 6. プロンプト設計ルール

AI生成機能を組み込む際のプロンプト設計指針。

### 基本構造

```
あなたは{役割}です。
以下の条件をもとに、{出力内容}を作成してください。

条件:
* テーマ: {theme}
* ターゲット: {target}
* 目的: {goal}
* 文体: {tone}
* その他パラメータ: {params}

出力ルール:
* {フォーマット指定}
* {品質基準}
* {制約条件}
```

### 品質を上げるコツ

- モデル性能だけに頼らず、プロンプト・ペルソナ・出力形式で改善する
- 出力パターン（型）を選択可能にする
- ペルソナ情報を毎回プロンプトに差し込む → 人格のブレを防ぐ

## 7. セキュリティルール

### 環境変数管理

```
ANTHROPIC_API_KEY=xxxx
NEXT_PUBLIC_SUPABASE_URL=xxxx
NEXT_PUBLIC_SUPABASE_ANON_KEY=xxxx
SUPABASE_SERVICE_ROLE_KEY=xxxx
```

- APIキーは絶対にクライアントへ出さず、サーバー側のみで使用
- `.env` ファイルは `.gitignore` に必ず追加
- `.env.example` にはキー名のみ記載（値は空）

### コーディング規約

- 外部入力は必ずバリデーション
- SQLはパラメータ化クエリを使用
- 依存パッケージは定期的にアップデート・脆弱性チェック
- HTTPSを使用（HTTPは開発環境のみ）

## 8. コミット・ブランチ規約

### コミットメッセージ

```
<type>: <概要>

<本文（任意）>
```

| Type | 用途 |
|------|------|
| `feat` | 新機能追加 |
| `fix` | バグ修正 |
| `docs` | ドキュメントのみの変更 |
| `style` | フォーマット等（コードの意味に影響しない） |
| `refactor` | リファクタリング |
| `test` | テストの追加・修正 |
| `chore` | ビルドプロセスや補助ツールの変更 |

### ブランチ命名

| ブランチ | 用途 |
|---------|------|
| `main` | 本番リリース用 |
| `develop` | 開発統合ブランチ |
| `feature/<名前>` | 新機能開発 |
| `fix/<名前>` | バグ修正 |
| `claude/<タスク名>` | Claude Code による自動開発 |

## 9. デプロイルール

### 手順

1. GitHubへpush
2. ホスティングサービス（Vercel等）に接続
3. 環境変数設定
4. デプロイ

### チェックリスト

- [ ] 環境変数がすべて設定されている
- [ ] APIキーがクライアントに露出していない
- [ ] ビルドが成功する
- [ ] 本番URLで動作確認

## 10. Claude Code スキル活用

プロジェクト固有のルールを `.claude/skills/` に保存して再利用する。

### 例: `.claude/skills/ui-saas-dark/SKILL.md`

```markdown
* ダークネオンで作る
* 余白広め
* 角丸大きめ
* 青紫発光
* SaaS風の高級感
* Lucide icons
* 再利用可能コンポーネント優先
```

毎回デザイン指示を繰り返さずに済む、プロジェクト専用の"制作マニュアル"として機能する。

## 11. よくある失敗と対策

| 失敗パターン | 対策 |
|-------------|------|
| 最初から外部API連携まで全部やる | まずUIと保存から。段階的に進める |
| Claude への指示が雑（「いい感じにして」） | 具体的な言葉で伝える |
| 1回で完成を狙う | 画面単位で区切って進める |
| DB設計を盛りすぎる | 最小構成から始める |
| AI品質をモデルだけで解決しようとする | プロンプト・ペルソナ・出力形式で改善 |
| 「なんか違う」で修正依頼 | 何が違うかを箇条書きで具体的に伝える |
